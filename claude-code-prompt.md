# Claude Code Prompt — "Agent Line": a reusable 2.5D agentic-orchestration storytelling engine

Copy everything below the line into Claude Code as your initial instruction. It is written to be self-contained.

---

## Project brief

Build a **reusable, config-driven web app** that tells the story of how information flows through a line of AI agents — a stylized "information assembly line." It is a **presentation/demo tool**, not a real integration: it visualizes an agentic-orchestration narrative for a live audience.

The visual style is **cartoon / stylized 2.5D** (flat illustrated shapes with depth via shadows, layering, and glow — like a polished animated explainer). **Do NOT use a real 3D engine.** Implement the scene in **HTML + CSS + SVG/Canvas with 2D animation**. It must run smoothly on an ordinary laptop with no GPU dependency.

The single most important architectural rule: **the engine contains zero story-specific content.** Everything about a given story — inputs, agent stations, outputs, the plot sequence, captions, and which media file attaches to each node — lives in a **JSON scenario config**. To tell a completely different story next time, the user writes a new JSON file and drops media into a folder. Nothing in the engine code should mention emails, WhatsApp, Autodesk, HDB, or any domain term. Those appear only in config and dummy data.

## Tech stack

- **Vanilla TypeScript + Vite** (no React, no heavy framework). Plain, easy to run and hand off.
- **No CDN / no external runtime dependencies** — everything bundled so it runs fully **offline** on `localhost`. Dev tooling (Vite) may come from npm, but the built app must not fetch anything at runtime.
- Rendering: **SVG for the crisp illustrated shapes** (machines, objects, agents), CSS for layout/animation, optional Canvas only if needed for particle/glow ambiance. Prefer SVG + CSS transitions/`requestAnimationFrame`.
- Run target: `npm run dev` on localhost. Also provide `npm run build` producing a static bundle that opens offline.

## Art generation — hard rule (read carefully)

**All visual assets are procedurally drawn in code as SVG, generated by you (Claude Code). There are NO external art files of any kind.** Specifically:

- **Do NOT** import, download, or reference any 3D models (glTF/GLB/OBJ), icon libraries/packs, font-icon sets, sprite sheets, or stock images for any visual element.
- **Do NOT** fetch any asset at runtime. The app is fully offline.
- Every object, badge, agent icon, machine, conveyor belt, input hopper, output crate, floor glow, and the **entire background scene** must be **hand-authored SVG/CSS drawn in code**. The backdrop is code-drawn too — no raster background image.
- The travelling information objects (envelope, chat bubble, card, photo, etc.) are SVG built from primitive shapes so they can be **animated and transformed** (badges are additional SVG layers composed on top at each station). This is why raster/imported art is forbidden — it can't be composed or morphed.
- The only place a user may *optionally* supply an external file is the **media slot inside a clicked popup** (their own screenshot/video), which is content, not scene art. Scene art is always code.

The upside to state plainly in your README: a new scenario needs **zero art work** — the primitive library already draws everything, and new stories are pure config.

## The scene (hardcoded backdrop, configurable content on top)

The **background is a fixed, hardcoded** deep-purple industrial "factory" scene. Match this aesthetic:
- Deep violet/purple gradient background (`#0b0618` → `#1e0f3d`), ambient radial glows in magenta/violet.
- A horizontal **conveyor belt** across the lower third with a glowing **cyan** (`#22e0ff`) leading edge and animated belt segments that scroll continuously.
- Stylized **machine housings** for each station (purple metal look, gradient `#4a2c8f` → `#25133f`, lighter top bevel), with an indicator light and a soft cyan glow beneath when active.
- **Agent chips** floating above each machine (rounded pills with an icon dot + agent name).
- An **input hopper** on the left and an **output crate** on the right (server/archive look).
- Accent palette: violet `#a100ff`, cyan `#22e0ff`, success green `#00e6a8`, amber `#ffb027`. Text in soft lilac-white.
- Overall: friendly, glowing, slightly playful "AI factory," readable by a non-technical senior audience.

The **number and placement of stations, and the labels/icons of everything, come from the config** — the backdrop art (belt, lighting, floor glow, hopper, crate) is the only hardcoded part. The engine lays out N stations evenly along the belt based on the active scenario.

## Node types (all defined in config)

1. **Inputs** — the raw information entering the line (e.g. an envelope, a chat bubble). A scenario may have **multiple inputs**. Each has: id, label, icon/shape, description, one media slot.
2. **Agent stations** — machines along the belt. Each has: id, station label (e.g. CAPTURE), one or more agent chips (name + icon), a short "what it does" description, the processing animation, one media slot, and the **object transformation** it applies (see below).
3. **Outputs** — the governed objects leaving the line (e.g. a ticket, a filed document, a classified photo). A scenario may have **multiple outputs**. Each has: id, label, icon/shape, description, one media slot.

**All three node types are clickable.** Clicking any node opens a **popup/modal** containing: the node's **title**, its **description**, and **one media slot** that displays an image OR plays a video, loaded from a file path in the scenario folder. If the media file is missing, show a tasteful placeholder ("media not attached") rather than breaking. Clicking a node **pauses autoplay**.

## Object transformation (important)

As the information object travels down the belt, it must **visibly change at each station** to show raw input becoming governed data. Example progression: plain envelope → envelope with coloured tags (after Understand) → ticket card (after Structure) → card with chain-link badge (after Link) → card with green "compliant" stamp (after Assure).

The transformations are **config-driven**: each station declares how it mutates the travelling object. Implement this as a set of **reusable visual "object states"** (composable SVG layers: base shape + optional badges/stamps/tags/colour) that the config references by name. The engine applies the named transformation when the object reaches that station. Provide a small library of generic, domain-neutral object primitives and badges (e.g. `base:envelope`, `base:chat`, `base:card`, `badge:tags`, `badge:link`, `badge:stamp-green`, `badge:rename`) so new scenarios can compose objects without new art.

## Primitive library — required manifest (`src/engine/primitives.ts`)

Build one module that draws every primitive as an SVG function. This is the entire art system; nothing else produces visuals. It **must** implement at least the following, because the seed scenarios and the provided HDB config reference them by name — any missing primitive would render blank:

- **Object bases** (`base:*`): `envelope`, `chat`, `card`, `photo`. Each is a clean, cartoon 2.5D shape with a subtle drop shadow and a highlight bevel so it reads with depth on the purple stage.
- **Object badges** (`badge:*`), drawn as small layers that compose on top of any base without overlapping each other (define fixed anchor slots — e.g. corners — so multiple badges stack cleanly): `tags` (small coloured label tags), `rename` (a little "renamed/label" tag or A→B glyph), `link` (a chain-link mark), `stamp-green` (a green circular "compliant" stamp).
- **Agent icons** (`icon:*`) used in the floating agent chips: `eye`, `tag`, `ticket`, `rename`, `link`, `check`, `shield`. Simple monoline glyphs drawn in SVG.
- **Scene furniture** (drawn by the engine, not config-referenced): conveyor belt with animated segments + glowing cyan edge, machine housing (with bevel, indicator light, active-glow), agent chip pill, input hopper, output crate/server, floor glow, ambient background gradient + radial glows.

Each primitive is a pure function returning an SVG group, parameterised by size/theme colours so it restyles centrally. Document the full list in `SCHEMA.md` and note clearly: **to add a new object type in future, add one function here — never an asset file.** Composition rule: a rendered object = one `base` + zero-or-more `badges` layered in a deterministic order/anchor so states like `["tags","rename","link","stamp-green"]` all show at once without collision.

## Plot / sequencing

- A scenario contains one or more **plots**. Each plot is a **linear sequence of steps**. A step references a node (input appears / object moves to station K / station processes / output emitted) and carries: a **caption** (shown as a timed subtitle), a **dwell time** (ms), and optionally which media to surface.
- Support **multiple inputs and multiple outputs** within a single plot (e.g. two inputs merge into the line; the line produces two distinct outputs at the end). Keep the data model **linear per object** but allow multiple objects/inputs and multiple outputs in one plot.
- Design the config so branching *could* be added later, but **do not implement branching now**.

## Camera / view

- **Fixed view of the whole line** (no camera travel). The active station **highlights** (brightens, glow ramps up, agent chips illuminate, processing ring pulses) while inactive stations dim slightly. Keep the entire line visible at all times.

## Controls & presentation

- **Autoplay** — steps advance automatically using each step's dwell time, with **timed captions** appearing as subtitles.
- **Manual mode** — step forward/back with **arrow keys**; **spacebar** toggles play/pause.
- **Scenario switcher** — a nav pane / dropdown to load a different scenario JSON at runtime (e.g. switch between "Email" and "WhatsApp").
- **Plot switcher** — if a scenario has multiple plots, allow choosing which to play.
- **Reset** button — return to the start of the current plot.
- **Fullscreen / kiosk** toggle for the venue.
- A **navigation pane** showing the plot steps as a clickable list (jump to any step); highlight the current step.
- Respect `prefers-reduced-motion`.
- Everything keyboard-accessible; visible focus states.

## Quality bar

- Smooth 60fps-ish animation on an ordinary laptop; degrade gracefully.
- No runtime network calls; fully offline.
- Domain-neutral engine — verify by confirming the two seed scenarios differ *only* in their config + media, not in engine code.
- **All scene art is code-drawn SVG** — no imported models, icon packs, or image files anywhere except user-supplied popup media. Verify the primitive library covers every `base`, `badge`, and `icon` referenced by any provided config, so nothing renders blank.
- Clean, readable TypeScript with the engine's public API documented.
- Presentation-hardened: fullscreen, keyboard control, reset, missing-media placeholders, malformed-config messaging.
